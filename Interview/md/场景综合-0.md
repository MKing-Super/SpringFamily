## 1 SpringCloud在本地自测

**方式一：Zuul**

spring-boot 的 1.5.10.RELEASE 版本：

```
客户端（手机、PC）
    ↓
[ Ngix ] ←→ 反向代理
    ↓
[ Zuul API Gateway ] ←→ 网关：负载均衡、路由、过滤
    ↓
[  Eureka Server  ] ←→ 服务注册中心：服务发现
    ↑        ↑
A-admin     A-admin    ←→ 微服务实例
(注册)       (注册)
    ↑         ↑
B-service ←→ B-service    ←→ service站点相互调用
(注册)       (注册)
```

**原理：**

​		**Ribbon 是 Zuul 的负载均衡引擎**。Zuul 依赖 Ribbon 来实现服务的负载均衡调用。 

```
客户端请求
    ↓
[  Zuul Gateway  ] ← 接收请求
    ↓
路由匹配 → 找到目标服务
    ↓
[  Ribbon 负载均衡  ] ← 从 Eureka 获取服务列表
    ↓
选择具体服务实例
    ↓
转发请求
```

**实现方法：**

A-admin：

```yml
#  拒绝从eureka中获取实例
ribbon.eureka.enabled = false
# 指定 zuul 网关服务
zuul-service.ribbon.listOfServers = http://127.0.0.1:9000
```

Zuul：

​		不向集群的eureka注册，只向eureka中获取其他站点的接口信息，避免影响其他集群项目的站点使用本地启动的zuul。

​		将zuul路由指向成本地的启动的站点。

```yml
eureka:
  client:
    serviceUrl:
      defaultZone: http://{集群中的IP:port}/eureka/
    register-with-eureka: false	# 是否向eureka注册

zuul:
  routes:
    A-service:
      path: /A-service/**
      url: http://127.0.0.1:9001	# 本地启动的A-service
    B-service:
      path: /B-service/**
      url: http://127.0.0.1:9002	# 本地启动的B-service
```

Host：

​		配置host使得域名指向本地IP。admin项目使用默认端口80启动。

```bash
127.0.0.1	a.test.com
```



**方式二：微服务站点直连**

spring-boot 的 2.3.12.RELEASE 版本：

原理：**Ribbon 的静态服务列表配置**

```
┌─────────────┐    1. 读取配置     ┌─────────────┐
│             │◄──────────────────│             │
│  Ribbon     │                   │ application │
│  Client     │                   │   .yml      │
│             │                   │             │
└──────┬──────┘                   └─────────────┘
       │
       │ 2. 创建 ConfigurationBasedServerList
       │
┌──────▼──────┐    3. 获取服务列表 ┌─────────────┐
│ ServerList  │───────────────────►│  配置中的    │
│  Interface  │                   │ listOfServers│
└──────┬──────┘                   └─────────────┘
       │
       │ 4. 返回服务实例列表
       │
┌──────▼──────┐
│ LoadBalancer│
│             │
└──────┬──────┘
       │
       │ 5. 选择服务实例
       │
┌──────▼──────┐
│  发起HTTP    │
│   请求      │
└─────────────┘
```

**实现方法：**

A-admin（单个service）

```yml
# A-admin 指向 本地的home-service
home-service:
  ribbon:
    listOfServers: http://localhost:9005
    NIWSServerListClassName: com.netflix.loadbalancer.ConfigurationBasedServerList
```

> 各个部分的作用：
>
> **`listOfServers`**：
>
> - 指定服务的实例地址列表
> - 格式：`host:port,host2:port2`
> - 多个地址用逗号分隔
>
> - 这是**硬编码的服务地址**，不依赖服务注册中心
>
> **`NIWSServerListClassName`**：
>
> - 指定 Ribbon 使用的服务列表实现类
> - `ConfigurationBasedServerList`表示从配置文件读取服务列表
> - 这是 Netflix Internal Web Service Server List 的缩写

A-admin（多个服务实例）

```yml
home-service:
  ribbon:
    # 多个实例，Ribbon会自动轮询
    listOfServers: http://localhost:9005,http://localhost:9006,http://localhost:9007
    NIWSServerListClassName: com.netflix.loadbalancer.ConfigurationBasedServerList
    # 负载均衡策略
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RoundRobinRule
```



## 2 Spring自定义注解的使用

​		在Spring中自定义注解是一种强大的元编程手段，用于简化配置、增强可读性和实现声明式编程。

使用步骤：

1 .定义注解

使用`@interface`关键字创建注解类，可指定元注解（如`@Target`、`@Retention`）控制其使用范围和生命周期。

```java
import java.lang.annotation.*;

@Target(ElementType.METHOD)      // 注解作用于方法
@Retention(RetentionPolicy.RUNTIME) // 运行时保留
public @interface MyCustomAnnotation {
    String value() default "";    // 注解参数
    int priority() default 0;
}
```

2 .使用注解

```java
@Service
public class UserService {
    
    @MyCustomAnnotation(value = "测试操作", priority = 1)
    public void processData() {
        // 业务逻辑
    }
}
```

3 .处理注解逻辑

通过以下方式实现注解的解析和行为注入：

**方式一：AOP拦截（常用）**

结合Spring AOP，在切面中识别注解并增强逻辑。

```java
@Aspect
@Component
public class MyAnnotationAspect {
    
    @Around("@annotation(myAnn)") // 拦截带有MyCustomAnnotation的方法
    public Object aroundAdvice(ProceedingJoinPoint joinPoint, 
                               MyCustomAnnotation myAnn) throws Throwable {
        System.out.println("注解参数: " + myAnn.value());
        // 前置处理
        Object result = joinPoint.proceed(); // 执行原方法
        // 后置处理
        return result;
    }
}
```

**方式二：BeanPostProcessor**

在Bean初始化阶段扫描注解并处理。

**方式三：实现ApplicationListener**

监听应用事件，在特定阶段处理注解。



**典型应用场景**

- **日志记录**：自动记录方法入参、执行时间。





## 3 springboot在jdbc连接mysql的最大连接数为20时，此时有40个事务同时并发。那40个中只有一半的事务会完成，剩下全部抛异常吗？

​		SpringBoot + JDBC 连接 MySQL 时，连接池会通过**等待队列**和**超时机制**来处理超出最大连接数的请求，而非直接拒绝，最终结果取决于你对连接池的超时配置。

**1 .连接池的 “排队机制”**

当你设置 MySQL 连接池最大连接数为 20 时，连接池的工作流程是这样的：

1. **前 20 个事务**：直接获取空闲连接，正常执行；
2. **第 21-40 个事务**：进入**等待队列**，等待前面的事务释放连接；
3. **等待超时 / 队列满**：只有等待时间超过阈值，或等待队列也满了，才会抛出异常。



**2 .关键配置**

HikariCP 是 SpringBoot 默认的连接池，核心配置决定了 40 个并发的最终结果：

```yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20       # 最大连接数（核心）
      minimum-idle: 5             # 最小空闲连接
      connection-timeout: 30000   # 获取连接的超时时间（默认30秒）
      idle-timeout: 600000        # 连接空闲超时时间
      max-lifetime: 1800000       # 连接最大存活时间
      queue-capacity: 50          # 等待队列容量（部分连接池如Tomcat JDBC有此配置，HikariCP无显式队列，靠timeout控制）
```

**3 .40 个并发事务的 3 种可能结果**

**场景 1：事务执行速度快（最优）**

- 前 20 个事务在**30 秒内**完成并释放连接；
- 等待队列中的 20 个事务依次获取连接，全部正常执行；
- **结果**：40 个事务都完成，无异常。

**场景 2：事务执行较慢，但未超时**

- 前 20 个事务执行时间超过几秒，但在 30 秒内；
- 等待队列中的事务会**阻塞等待**（不会抛异常），直到拿到连接；
- **结果**：40 个事务都完成，但整体响应时间变长（后 20 个需要等）。

**场景 3：事务执行超时（最常见异常场景）**

- 前 20 个事务执行时间超过 30 秒（`connection-timeout`）；

- 等待队列中的事务等待 30 秒后仍拿不到连接，抛出

  连接超时异常

  ```
  com.zaxxer.hikari.pool.HikariPool$PoolTimeoutException: 
  Timeout after 30000ms waiting for a connection from the pool.
  ```

- **结果**：前 20 个事务正常完成，后 20 个抛出超时异常（而非 “直接拒绝”）。

**4 .如何优化这种场景？**

1. **合理调大最大连接数**：根据 MySQL 性能调整（MySQL 默认最大连接数是 151，可设`maximum-pool-size: 50`，但不宜过大，避免 MySQL 过载）；
2. **缩短事务执行时间**：优化 SQL、减少事务内的非数据库操作（如 RPC 调用、IO）；
3. **设置合理超时**：根据业务场景调整`connection-timeout`（比如短事务设 5 秒，避免长时间阻塞）；
4. **添加熔断 / 限流**：通过 Sentinel、Hystrix 等限制并发数，避免连接池被打满。
